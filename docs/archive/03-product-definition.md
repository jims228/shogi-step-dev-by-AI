# プロダクト定義

## 誰のためのMVP
- **将棋初心者〜入門者**（駒の動かし方から学びたい人）
- 日本語メインだが、将来的に英語対応も視野

## MVPができること
1. **レッスン一覧**を表示する
2. 選んだ**レッスンの詳細**を表示する
3. レッスン内の**ステップを順に進める**（説明 → 盤面 → クイズ）
4. **説明ステップ**: テキスト＋画像のレンダリング
5. **盤面ステップ**: SFEN文字列からの最小限の盤面表示（駒の位置がわかるレベル）
6. **クイズステップ**: 選択式クイズの表示・正誤判定
7. **進捗の保存**: localStorage による最終到達レッスン/ステップの記録

## 明示的にスコープ外
- ユーザー認証・アカウント
- サーバーサイド・データベース
- 課金・決済
- AI によるランタイム生成
- 対局機能
- 高度な盤面操作（ドラッグ＆ドロップ、アニメーション）
- 多言語対応（v1では日本語のみ）
- PWA / モバイルアプリ

## 技術選定: Vite vs Next.js

**結論: Vite + React を採用**

| 観点 | Vite | Next.js |
|------|------|---------|
| SSR/SEO | 不要（学習アプリ、検索エンジン対応不要） | 主要利点がこのプロダクトでは活きない |
| バンドルサイズ | 軽量 | App Routerの追加コスト |
| 複雑性 | 最小（SPA） | サーバーコンポーネント/クライアントコンポーネントの区別が不要な複雑性 |
| デプロイ | 静的ホスティング（GitHub Pages/Vercel） | サーバーが必要（or 静的エクスポート） |
| API | 不要（全てクライアントサイド） | API Routes不要 |
| 旧資産 | commentary-aiはNext.jsだが、その複雑性は不要 | — |

このプロダクトはDB不要・API不要・SSR不要の**純粋なクライアントサイドSPA**。Viteが最適。

## 最初の3レッスンの最適化方針

| レッスン | 最適化ポイント | 根拠 |
|---------|--------------|------|
| **01: 将棋ってなに？** | **恐怖心の除去**。将棋は難しそうという先入観を壊す。盤面を見せて「こんなゲームです」だけ。ルール詳細には入らない | mobile旧アプリのrules系レッスン11本のうち、最初が概要紹介だった |
| **02: 歩の動き** | **最小限の成功体験**。1つの駒だけに集中。「歩が前に1マス動ける」をクイズで確認し「できた！」を体感させる | mobile `basics_pawn.ts` が4ステップで歩を教えていた。成駒は後にする |
| **03: 金将の動き** | **パターン認識の導入**。歩（1方向）→ 金（6方向）で「駒によって動きが違う」という概念を定着させる。王の守り役として文脈を与える | mobile `basics_gold.ts` の構成を参考に |

**3レッスン共通原則**:
- 1ステップ = 1概念（詰め込まない）
- 各レッスン末尾に2つのクイズ（自信を育てる）
- SFEN盤面は**駒1〜2個の極小配置**（初期配置は01のみ）
- 励ましのトーンを統一（commentary-aiの`encouraging`スタイル準拠）
